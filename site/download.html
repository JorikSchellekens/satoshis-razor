Satoshi's Razor

Get started

One script builds the whole stack — the Lean package, the registry CLI, the benchmark harness, the Groth16 prover — and puts the tools on your PATH. That is all it does: it writes no dataset and starts no server. Hosting the registry locally is one command afterwards.

Install

curl -sSf http://localhost:8420/install.sh | bash # or, from a clone of the repository: git clone https://github.com/jorikschellekens/satoshis-razor cd satoshis-razor && ./install.sh

The script checks for git and rust (via rustup); installs elan (the Lean toolchain manager) and the wasm32 compile target if they are missing; builds everything; and links the three tools into ~/.cargo/bin. Read the script before you run it — you should not have to trust this page's word about what it does. The source of everything, this site included, is github.com/jorikschellekens/satoshis-razor.

Host the registry locally

razor serve # browse it at http://localhost:8420

The clone already contains the live log (registry/data/events.jsonl), so there is nothing to load: razor serve re-derives data.json from it on every request - append an event from the CLI and every open page updates within a few seconds. To see the scripted walkthrough with fictional participants instead, run ./demo.sh first (and restore afterwards with git checkout registry/data/events.jsonl).

What you have afterwards

Three commands on your PATH (linked into ~/.cargo/bin, so a rebuild updates them in place), and two files that are the system's entire state:

razor # the registry CLI: append events, verify proofs, serve the site anvil-harness # fuel + native benchmarks zk-prover # Groth16 setup / prove / verify, any list length 2-32 registry/data/events.jsonl # the append-only log - the registry IS this file site/data.json # everything above, folded into one snapshot for these pages

Get your events onto the public registry

the hosted site serves the log on the repository's master branch

Every razor command appends to your local copy of registry/data/events.jsonl. The public registry at razor.mempoolsurfer.com serves the copy of that file on the master branch of github.com/jorikschellekens/satoshis-razor. So publishing your work is a pull request:

# fork the repository on GitHub, then work in your fork razor account new # optional but recommended: your later events are signed # … razor seal-statement / formalize / submit / verify / … as documented on this page … git add registry/data/events.jsonl lean/ lean-mathlib/ git commit -m "my events" # the appended events + any Lean files they reference # open a pull request against master

CI re-checks every signature on the log for every pull request, and a maintainer re-runs the kernel checks (razor recheck) before merging. Once the pull request merges, the hosted site updates within a few minutes. Your position on the shared log is your events' order after the merge — for sealed readings that is the order that the blindness facts are computed from, so a seal establishes independence as long as it merges before the other readings' reveals.

Two things to avoid in a pull request. Do not commit a log produced by ./demo.sh — that is the fictional walkthrough dataset. And do not run ./seed.sh before contributing — it regenerates the whole log from scratch with fresh timestamps and signatures, so your diff would rewrite history instead of appending to it. The clone already contains the live log; append to it as it is. If you ran either script, restore first: git checkout registry/data/events.jsonl.

Proofs can also live in a new file: put any .lean file under lean/Razor/ (or lean-mathlib/RazorMathlib/ for Mathlib-environment holes) - the build picks it up automatically, and the verifier finds the declaration wherever it is defined.

# check your proof builds cd lean && lake build && cd .. # submit and verify (the kernel is the referee) razor submit --id SUB-me --hole RZR-104 \ --solver you --decl Razor.Sorting.merge_sorted razor verify --submission SUB-me # regenerate the site razor export --out site/data.json

Prefer to work in private?

# commit a hash first (priority without exposure) … razor seal --file my-proof.lean --salt my-secret razor commit --id SUB-me --hole RZR-104 --solver you --commitment <hash> # … reveal when ready razor reveal --submission SUB-me --file my-proof.lean \ --salt my-secret --decl Razor.Private.You.merge_sorted # or go full zero-knowledge: solve through a hole's zk route (see the zk page) zk-prover prove --list 42,7,255,7 razor zk-submit --id SUB-me-zk --hole ZKH-001 --route ZKR-001 \ --solver you --public <hex> --proof <hex>

Work on the live registry

The live registry (./seed.sh) carries real open problems from the Lean community's current frontier. Most are still proposals waiting for a formal statement; the Erdos-Straus conjecture already has one pinned hole per environment (ES-001 in core Lean, ES-002 in Mathlib's definitions), and Fermat's Last Theorem is pinned in core Lean (FLT-000) with two registered splits whose glue proofs are already admitted. FLT-RED - every exponent at least 3 factors through 4 or an odd prime - is self-contained number theory and a genuinely provable first hole on the live board.

Add your own candidate statement with razor formalize --gloss "..." - the gloss is your plain-language reading, which is what reviewers compare against the proposal - and strengthen a clump by proving your statement equivalent to an existing one (razor converge).

Share partial progress as a split

If you can prove a hole from lemmas you cannot yet prove - in a Lean file, a proof with sorry in the gaps - register the gaps as holes and the reduction as a split. The CLI composes the glue statement mechanically from the pinned types, so there is nothing to get subtly wrong:

# state each gap as its own hole … razor hole --id MY-LEMMA-1 --title "..." --lean-type "..." # … tie them to the parent; this pins MY-SPLIT-glue to (lemma 1) → … → parent razor split --id MY-SPLIT --parent RZR-104 --author you \ --child MY-LEMMA-1 --note "induction on the left list" # prove the glue like any hole - checkable while every child is still open razor submit --id SUB-glue --hole MY-SPLIT-glue --solver you --decl You.my_glue razor verify --submission SUB-glue

When all children and the glue are admitted, the split reads complete: the parent is provable by composition, which in your original file is just replacing each sorry with the admitted proof. If the plan turns out wrong, register a better split next to it - splits are never edited, because an admitted glue is a true theorem either way.

Accounts, signatures, and the sandbox

razor account new generates an Ed25519 keypair: the public key goes on the log as your handle's identity, the signing key stays on your machine (registry/data/keys/). From then on every event you author is signed automatically, the CLI refuses to append events in a registered handle's name without its key, and razor verify-log re-checks every signature on the log. Handles without accounts can still do everything - their events are simply marked as unsigned.

Verification treats a submission as untrusted code, because Lean elaboration can run programs: the checker runs with network access denied (sandbox-exec on macOS, bwrap on Linux) and is killed after a time limit. Linux does not ship bwrap by default — install it with sudo apt install bubblewrap; without it the checker warns and runs unsandboxed.

Proofs that use Mathlib

Holes registered with --env mathlib pin statements written with Mathlib's definitions and are verified in the lean-mathlib/ package, which requires Mathlib v4.31.0 (matching the pinned toolchain, so the prebuilt cache applies). Fetch it once with ./mathlib-env.sh - it is several gigabytes. Until then the registry refuses to verify Mathlib-environment submissions and says why.

Enter the anvil

Write a faster implementation of a challenge spec in anvil/impls/, hand-translate it to a Lean model in lean/Razor/Anvil/, prove it refines the executable spec, then razor anvil-submit and razor bench. Your lane appears on the leaderboards next to its proof.

Bring a rig in Docker

Native scores name the machine they were measured on, and a rig can be a container instead of bare hardware: it registers a runner command, and razor bench --rig executes the harness through it. One script builds a Linux image with the harness inside, registers it, and measures every challenge in it — on a Mac this means real Linux scores (Docker runs containers in a Linux virtual machine) appear on the boards next to your host's native scores:

./demo.sh # the challenges (skip if already seeded) ./docker-rig.sh # build image, register rig docker-linux-<arch>, bench everything in it
